# 07. 動作環境（本番サーバ・外部サービス）構築手順

本書は **誰がやっても同じ手順で構築できる** ことを目的に、各サービスの GUI 操作をクリック単位で記載する。

> 💡 各章末尾に「動作確認」が付いているので、手順完了後は必ず確認してから次に進むこと。
> 💡 手順中の画像キャプションは執筆時点（2026-04）の UI に基づく。UI 変更があった場合はボタン名で読み替え可能な形にしている。

---

## 0. 全体像

| Step | サービス | 所要時間 |
|---|---|---|
| 1 | Firebase プロジェクト作成 + Firestore 有効化 + サービスアカウント | 約 20 分 |
| 2 | LINE Developers チャネル作成 + LIFF 登録 | 約 20 分 |
| 3 | Stripe アカウント + 決済方法 + Webhook | 約 30 分 |
| 4 | RemoteLock API 発行 + デバイス ID 取得 | 約 15 分 |
| 5 | Vercel プロジェクト作成 + 環境変数投入 + デプロイ | 約 30 分 |

全所要時間: **約 2 時間**（アカウント事前登録済みの場合）。

## 1. Firebase / Firestore 構築

### 1.1 プロジェクト作成

1. ブラウザで [https://console.firebase.google.com/](https://console.firebase.google.com/) を開く
2. Google アカウントでログイン
3. 画面中央の **「プロジェクトを追加」** をクリック
4. **プロジェクト名**: `gym-checkin-prod`（任意の名前で可）を入力 →「続行」
5. **Google アナリティクス**: 使用しない場合はトグル OFF →「続行」または「プロジェクトを作成」
6. 作成完了を待つ（1〜2 分）→「続行」

### 1.2 Firestore Database 作成

1. 左サイドメニュー **「構築 (Build)」 → 「Firestore Database」** を選択
2. 画面中央の **「データベースの作成」** ボタンをクリック
3. **モード選択**: 「本番環境モードで開始する (Start in production mode)」を選択 →「次へ」
4. **ロケーション**: プルダウンから **`asia-northeast1 (東京)`** を選択 →「有効にする」
5. データベース作成が完了するまで待つ（30 秒程度）

### 1.3 セキュリティルールを全拒否に設定

1. Firestore Database 画面 → 上部タブ **「ルール (Rules)」**
2. エディタの内容を以下で置換:

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /{document=**} {
      allow read, write: if false;
    }
  }
}
```

3. 画面右上 **「公開 (Publish)」** をクリック
4. 確認ダイアログで **「公開」** を押下

> 📌 **理由**: 本アプリはサーバー（Vercel Functions）経由のみで Firestore にアクセスするため、クライアント直アクセスは全拒否する。

### 1.4 サービスアカウント発行

1. 画面左上の **歯車アイコン** → **「プロジェクトの設定」** をクリック
2. 上部タブ **「サービス アカウント」** を選択
3. **「新しい秘密鍵の生成」** ボタンをクリック
4. 確認ダイアログで **「キーを生成」** をクリック → JSON ファイルがダウンロードされる
5. ダウンロードしたファイルを **`firebase-service-account.json`** にリネーム（任意）
6. ターミナルで Base64 化してクリップボードにコピー:

```bash
# macOS
cat firebase-service-account.json | base64 | tr -d '\n' | pbcopy

# Linux
cat firebase-service-account.json | base64 -w0 | xclip -selection clipboard
```

7. この Base64 文字列は後で Vercel の `FIREBASE_SERVICE_ACCOUNT_KEY` に貼り付ける → **一時的にメモ帳等へ保存**

### 1.5 IAM 権限確認

1. [https://console.cloud.google.com/iam-admin/iam](https://console.cloud.google.com/iam-admin/iam) を開く
2. 画面上部のプロジェクト選択で先ほど作った Firebase プロジェクトを選ぶ
3. 一覧の中に `firebase-adminsdk-xxxxx@<project>.iam.gserviceaccount.com` があることを確認
4. ロール欄に **「Cloud Datastore User (Cloud Datastore ユーザー)」** が含まれていればOK。無ければ:
   - 行の **鉛筆アイコン (編集)** をクリック
   - **「ロールを追加」** → **「Cloud Datastore ユーザー」** を選択 →「保存」

### 1.6 動作確認

- Firebase Console → Firestore Database → **「データ」タブ** で「コレクションが空です」と表示されていれば成功

## 2. LINE Developers（LIFF）構築

### 2.1 プロバイダー作成（初回のみ）

1. [https://developers.line.biz/console/](https://developers.line.biz/console/) を開く
2. LINE アカウントでログイン
3. 画面右上 **「コンソールに入る」**
4. 左上 **「プロバイダー」 → 「作成」**
5. プロバイダー名: `if(塾)` 等を入力 →「作成」

### 2.2 LINE Login チャネル作成

1. 作成したプロバイダーを選択
2. **「新規チャネル作成」**
3. **「LINE Login」** を選択
4. 入力:
   - **チャネル名**: `みんなの体育館 予約`
   - **チャネル説明**: 任意
   - **アプリタイプ**: **「ウェブアプリ」にチェック**
   - その他必須項目（プライバシーポリシー URL 等）を入力
5. 利用規約に同意してチェック →「作成」

### 2.3 LIFF アプリ追加

1. 作成したチャネル画面を開く
2. 上部タブ **「LIFF」** を選択
3. **「追加」** ボタンをクリック
4. 入力:
   - **LIFF アプリ名**: `みんなの体育館`
   - **サイズ**: **`Full`**
   - **エンドポイント URL**: `https://<本番URL>/`
     - まだ Vercel に本番 URL が無い場合は仮で `https://example.com/` を入れて、Vercel デプロイ後に更新する
   - **Scope**: ✅ `profile` ✅ `openid`（必須）
   - **ボットリンク機能**: **`On (Aggressive)`** 推奨
   - **Scan QR**: OFF
   - **モジュールモード**: OFF
5. チェック項目を確認 →「追加」
6. 作成された **LIFF ID**（`2008793930-iDPTCREA` のような文字列）を控える → Vercel の `VITE_LIFF_ID` に使用

### 2.4 チャネル ID / シークレット取得（将来の Messaging API 連携用）

1. 上部タブ **「チャネル基本設定」**
2. **チャネル ID / チャネルシークレット** をメモ

### 2.5 動作確認

- LIFF URL: `https://liff.line.me/<LIFF_ID>` を LINE 内ブラウザで開く
- 現段階では エンドポイント URL が仮なら白画面で OK（後で Vercel デプロイ URL に差し替え）

## 3. Stripe 構築

### 3.1 アカウント登録

1. [https://dashboard.stripe.com/register](https://dashboard.stripe.com/register) を開く
2. メールアドレス・パスワードで登録
3. メール認証を完了

### 3.2 本番モード有効化（本稼働前）

1. ダッシュボード左上の **「本番環境を有効化」** リンクをクリック
2. 法人・事業情報（屋号・事業者情報・口座）を入力
3. 本人確認書類をアップロード
4. 審査通過まで 1〜3 営業日
5. 審査完了後、左上のトグルで **「テスト環境」/「本番」** 切替可能

> 🧪 検証フェーズでは「テスト環境」のまま進めて OK。

### 3.3 API キー取得

1. 左メニュー **「開発者」 → 「API キー」**
2. **シークレットキー** の「+ キーを作成」または「表示」→ `sk_test_...`（テスト）／`sk_live_...`（本番）をコピー
3. 後で Vercel の `STRIPE_SECRET_KEY` に貼る

### 3.4 決済方法の有効化（カード／PayPay）

1. 左メニュー **「設定」 → 「決済方法 (Payment methods)」**
2. **カード**: デフォルトで有効
3. **PayPay**（日本のみ）:
   - 一覧の「PayPay」行の **「有効にする」** をクリック
   - 同意チェック →「送信」
   - 審査完了まで数営業日
4. 同じ画面で「Checkout で有効な決済方法」を確認

> 📄 PayPay 有効化の詳細は [`../paypay有効化/`](../paypay有効化/) フォルダ参照。

### 3.5 Webhook 登録

1. 左メニュー **「開発者」 → 「Webhook」**
2. **「エンドポイントを追加 (Add endpoint)」** をクリック
3. 入力:
   - **エンドポイント URL**: `https://<Vercel本番URL>/api/payments/stripe-webhook`
     - Vercel デプロイ完了後に設定（この時点では仮でもOK、後で編集可）
   - **リッスンするイベント**: 「+ Events を選択」→ **`checkout.session.completed`** のみチェック
4. **「エンドポイントを追加」** をクリック
5. 作成されたエンドポイントの **「署名シークレット (Signing secret)」** → **「表示」** をクリックし `whsec_...` をコピー
6. 後で Vercel の `STRIPE_WEBHOOK_SECRET` に貼る

### 3.6 動作確認

- 左メニュー「Webhook」→ 作成したエンドポイント → **「イベントを送信 (Send test webhook)」**
- 任意のイベントで「送信」→ **200 レスポンス** が返ればサーバー側で受信できている（Vercel デプロイ完了後に実施）

## 4. RemoteLock 構築

### 4.1 API 利用申請

1. 日本国内は代理店経由（[https://www.remotelock.jp](https://www.remotelock.jp) から問い合わせ）
2. API 開発者アカウントの発行依頼を送付（用途: 「会員向け予約システムからの PIN 発行」）
3. 発行されたログイン URL: [https://app.remotelock-pf.jp](https://app.remotelock-pf.jp)

### 4.2 OAuth Client（Client Credentials）発行

1. 管理コンソールにログイン
2. 左メニュー **「API クライアント」** → **「新規作成」**
3. 入力:
   - **名称**: `gym-checkin`
   - **グラントタイプ**: `Client Credentials`
   - **スコープ**: 画面に表示される項目のうち、予約作成（bookings 系）・デバイス参照（devices 系）に関するものを **全てチェック**。UI 表記は代理店の発行時に従う（`bookings.*` / `devices.*` 相当）
4. **「作成」** → 発行された **Client ID / Client Secret** をコピー
5. 後で Vercel の `REMOTELOCK_CLIENT_ID` / `REMOTELOCK_CLIENT_SECRET` に貼る

### 4.3 デバイス ID の取得

1. 左メニュー **「デバイス」**
2. 該当デバイスをクリックして詳細画面へ
3. URL の末尾 `/devices/<DEVICE_ID>` の部分をコピー、または詳細画面に表示されている **「デバイス ID」** をコピー
4. 4 デバイス全てについて以下を記録:

| 拠点 | 施設 | 環境変数 | シリアル例 | 取得した Device ID |
|---|---|---|---|---|
| ASP | 玄関 | `REMOTELOCK_DEVICE_ID_ENTRANCE` | `AC000W017827101` | ※記録 |
| ASP | 体育館 | `REMOTELOCK_DEVICE_ID_GYM` | `AC000W017830299` | ※記録 |
| ASP | トレーニング | `REMOTELOCK_DEVICE_ID_TRAINING` | `AC000W017830548` | ※記録 |
| YABASE | 八橋 | `REMOTELOCK_DEVICE_ID_YABASE` | `AC000W017827205` | ※記録 |

詳細台帳は [`../SmartLockディバイス情報/`](../SmartLockディバイス情報/) 参照。

### 4.4 動作確認（OAuth 疎通）

```bash
curl -s -X POST "https://api.remotelock-pf.jp/oauth/token" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "client_id=$RL_ID&client_secret=$RL_SECRET&grant_type=client_credentials"
```

- `access_token` を含む JSON が返ればOK
- 401 の場合: Client ID/Secret 誤り

## 5. Vercel 構築（アプリ配信）

### 5.1 Vercel アカウント登録／ログイン

1. [https://vercel.com/signup](https://vercel.com/signup) を開く
2. **「Continue with GitHub」** で GitHub 連携ログイン
3. **Personal** プラン（Hobby）でOK（本アプリは Hobby プランの関数上限 12 個に収まる）

### 5.2 リポジトリのインポート

1. ダッシュボード右上 **「Add New…」 → 「Project」**
2. 初回は GitHub の連携画面で `ifJukuManager` リポジトリを選択肢に入るように **「Adjust GitHub App Permissions」**
3. 対象リポジトリを検索 → **「Import」**
4. **Configure Project** 画面:
   - **Project Name**: `gym-checkin`
   - **Framework Preset**: `Vite`（自動検出）
   - **Root Directory**: **「Edit」をクリック → `LINEMiniApp/gym-checkin` を選択**
   - **Build and Output Settings**: デフォルト（`vercel.json` が優先される）
   - **Environment Variables**: この時点では空のまま（次ステップで入力）
5. **「Deploy」** をクリック
6. 初回デプロイが始まる（失敗しても OK。環境変数未設定のため）

### 5.3 環境変数の投入

1. プロジェクト画面 → **「Settings」** タブ
2. 左メニュー **「Environment Variables」**
3. 以下をそれぞれ **Production / Preview / Development** 全てにチェックを入れて追加:

| Key | Value | 備考 |
|---|---|---|
| `VITE_LIFF_ID` | `2008793930-iDPTCREA` | §2.3 の LIFF ID |
| `FIREBASE_SERVICE_ACCOUNT_KEY` | Base64 文字列 | §1.4 でコピーした値 |
| `STRIPE_SECRET_KEY` | `sk_live_...` or `sk_test_...` | §3.3 |
| `STRIPE_WEBHOOK_SECRET` | `whsec_...` | §3.5 |
| `REMOTELOCK_CLIENT_ID` | §4.2 の Client ID | |
| `REMOTELOCK_CLIENT_SECRET` | §4.2 の Secret | |
| `REMOTELOCK_DEVICE_ID_ENTRANCE` | ASP 玄関 | §4.3 |
| `REMOTELOCK_DEVICE_ID_GYM` | ASP 体育館 | §4.3 |
| `REMOTELOCK_DEVICE_ID_TRAINING` | ASP トレーニング | §4.3 |
| `REMOTELOCK_DEVICE_ID_YABASE` | 八橋 | §4.3 |
| `ADMIN_LINE_USER_IDS` | `Uxxxx,Uyyyy` | 管理者 LINE userId。後述 §7 で取得 |
| `NEXT_PUBLIC_BASE_URL` | `https://<vercel-domain>` | Stripe redirect 用（後述 §5.4 で確定） |
| `SKIP_PAYMENT` | （未設定 or `false`） | 本番は未設定のままで OK。`true` を入れると Stripe をスキップして PIN のみ発行（テスト用） |
| `LOG_LEVEL` | `info` | 調査時は `debug` に変更 |
| `LINE_CHANNEL_ACCESS_TOKEN` | （任意） | 将来 LINE Messaging API で Push 通知を行う場合に設定。無くても予約機能は動作する |

**投入手順（1 件ずつ）**:
- **Key**: 変数名を入力
- **Value**: 値を貼り付け
- **Environment**: ☑ Production ☑ Preview ☑ Development すべてチェック
- **「Save」**

> ⚠️ `FIREBASE_SERVICE_ACCOUNT_KEY` は改行を含まない Base64 1 行で貼る。貼り付け後に改行が入ると復号に失敗する。

### 5.4 本番 URL の確認と関連設定の更新

1. プロジェクト Overview → 本番 URL（例: `gym-checkin.vercel.app`）を確認
2. 以下を更新する:
   - Vercel 側 `NEXT_PUBLIC_BASE_URL` = `https://gym-checkin.vercel.app`
   - LIFF チャネル画面の **「エンドポイント URL」** を `https://gym-checkin.vercel.app/` に変更
   - Stripe Webhook のエンドポイント URL を `https://gym-checkin.vercel.app/api/payments/stripe-webhook` に変更

### 5.5 再デプロイ

1. プロジェクト画面 → 上部 **「Deployments」** タブ
2. 最新デプロイの右端 `…` → **「Redeploy」**
3. **「Use existing Build Cache」** のチェックは外して **「Redeploy」**
4. ✅ 成功を確認（2〜3 分）

### 5.6 リージョン／関数タイムアウトの設定

**リージョン（実行リージョン）**:

1. プロジェクト → **「Settings」** → 左メニュー **「Functions」**
2. **「Region」** セクションを探す
3. 既定の `Washington, D.C., USA (iad1)` のままで OK（Fluid Compute は自動最適化）
4. 東京リージョン固定にしたい場合（Pro 以上）:
   - プルダウンから **`Tokyo, Japan (hnd1)`** を選択
   - **「Save」**

**関数タイムアウト**:

1. 同じ **「Functions」** 画面の **「Default Function Timeout」** を確認
2. 2026 年現在、全プラン既定 **300 秒** で問題なし（変更不要）
3. 個別関数でタイムアウトを変えたい場合は、`api/**.ts` 側に以下を追記:

```ts
// ファイル末尾
export const config = { maxDuration: 60 };  // 秒
```

### 5.7 カスタムドメイン（任意）

1. プロジェクト → **「Settings」** → **「Domains」**
2. 入力欄に `reserve.example.com` 等を入力 → **「Add」**
3. 指示通りに DNS（CNAME or A レコード）を追加
4. 数分〜数十分で有効化
5. 有効化後、**§5.4 と同じく LIFF / Stripe / `NEXT_PUBLIC_BASE_URL` を新ドメインに更新**

### 5.8 動作確認

1. LIFF URL (`https://liff.line.me/<LIFF_ID>`) を LINE 内で開く
2. 拠点選択画面が表示される
3. テスト予約 → Stripe テストカード `4242 4242 4242 4242` / 任意の有効期限・CVC で決済
4. 完了画面に 4 桁 PIN が表示される
5. RemoteLock 管理コンソールに `bookings` が作成されている
6. Stripe ダッシュボード → Webhook → Attempts でイベント 200 を確認
7. Vercel → **「Logs」** で `[OPERATION]` ログが出ていることを確認

## 6. 管理者 LINE userId の取得方法

管理画面 `/admin` にアクセスする LINE userId を特定する手順。

### 方法 A（推奨）: Vercel Logs から取得

1. 対象者に LINE から LIFF URL（公式アカウントのリッチメニュー等）を**一度起動してもらう**
2. Vercel ダッシュボード → 対象プロジェクト → **「Logs」** タブ
3. 画面上部の検索ボックスに `user.autoRegister.ok` または `checkin.create` を入力
4. 直近のログに含まれる `lineUserId` フィールドの値（`U...` で始まる文字列）が対象者の userId

### 方法 B: Firestore から取得

1. 対象者に LIFF を一度起動してもらう（= `users` コレクションに自動登録される）
2. Firebase Console → Firestore Database → `users` コレクションを開く
3. `displayName` で対象者のドキュメントを探し、`lineUserId` フィールドをコピー

### userId の反映

4. 取得した userId を Vercel の `ADMIN_LINE_USER_IDS` にカンマ区切りで追加

```
ADMIN_LINE_USER_IDS=Uab12cd34ef56,U99aabb77cc33
```

5. Vercel で環境変数を更新 → 次回関数実行時に自動反映される。即時反映したい場合は **Deployments → 最新の `…` → Redeploy**

## 8. パラメータ設定 まとめ（本番）

| 設定項目 | 値 / 場所 |
|---|---|
| Node.js バージョン | 20.x（`package.json engines` / Vercel 自動認識） |
| Build Command | `npm run build`（`vercel.json`） |
| Output Directory | `dist`（`vercel.json`） |
| Install Command | `npm ci`（`vercel.json`） |
| SPA Fallback | `vercel.json rewrites`（`/((?!api/).*)` → `/index.html`） |
| CORS | `vercel.json headers` |
| 関数実行リージョン | 既定（`iad1`）または東京（`hnd1`）※Pro 以上 |
| 関数タイムアウト | 既定 300 秒（個別は `maxDuration` で上書き） |
