# 02. 詳細設計書

## 1. 画面設計

### 1.1 画面遷移

```
LocationPage (/)
  └─▶ HomePage (/facility)
        └─▶ CheckinPage (/checkin)
              └─▶ PaymentPage (/payment)
                    └─▶ Stripe Checkout (外部)
                          └─▶ CompletePage (/complete)  ──▶ ReviewPage (/review)

サイドフロー:
  ReservationsPage (/reservations)
  EventListPage (/events) ─▶ EventDetailPage (/events/:id)
  SchoolListPage (/schools) ─▶ SchoolDetailPage (/schools/:id)
  AdminPage (/admin)  (認可: lineUserId が ADMIN_LINE_USER_IDS に含まれる場合)
  PrivacyPolicyPage (/privacy) / CommerceLawPage (/commerce-law)
```

### 1.2 ルーティング定義

| ルート | コンポーネント | 認可 |
|---|---|---|
| `/` | `LocationPage` | LIFF |
| `/facility` | `HomePage` | LIFF |
| `/checkin` | `CheckinPage` | LIFF |

### 1.4 共通仕様

#### 1.4.1 LIFF 初期化 / 認可（`src/App.tsx`）

| イベント | トリガー | 仕様 |
|---|---|---|
| アプリ起動 | `App` マウント時 | `initializeLiff()` を呼び出し、LIFF SDK を `VITE_LIFF_ID` で初期化 |
| 自動ユーザー登録 | LIFF 初期化完了後 | `userApi.getMe()` を投げ、失敗しても UI はブロックしない |
| 初期化失敗（本番） | `initializeLiff` reject | 画面に「LIFFの初期化に失敗しました」＋「再読み込み」ボタンを表示（`window.location.reload()`） |
| 初期化失敗（開発） | 同上 | モックユーザー `U_dev_user_12345` で継続 |
| ルーティング | 初期化成功後 | `BrowserRouter` で各 `Route` をマウント |

#### 1.4.2 共通 UI コンポーネント

| コンポーネント | トリガー / props | 仕様 |
|---|---|---|
| `Header` | `showBack` | `true` のとき `history.back()` ボタンを表示 |
| `Button` | `disabled` / `onClick` | `disabled=true` のとき CSS でグレー化しイベントを受けない |
| `Loading` | `fullScreen` | 画面全面のスピナー |
| `DebugPanel` | 画面下ゲージを長押し | `debugStore` のスキップフラグを切替（本番でも有効） |

#### 1.4.3 状態管理（Zustand）

| Store | 永続化 | ライフサイクル |
|---|---|---|
| `checkinStore` | なし（メモリのみ） | `LocationPage` マウント時に `reset()` で初期化、`CompletePage` 成功時にもリセットされる |
| `debugStore` | localStorage（`gym-checkin-debug`） | セッションをまたいで保持 |

#### 1.4.4 エラー表示ルール

- API 呼び出し失敗: 呼出し元 `useState<string \| null>` にメッセージを保持し、インライン赤帯で表示
- 409 競合（重複予約・定員）: エラーメッセージをそのまま表示
- その他 500: 「決済処理中にエラーが発生しました。もう一度お試しください。」等の汎用メッセージ

#### 1.5.3 CheckinPage（`/checkin`）日時選択

| # | イベント | トリガー | 処理内容 |
|---|---|---|---|
| 1 | 初期表示 | マウント | `location` / `facilityType` のどちらかが未選択なら `/` に戻る |
| 2 | 日付候補生成 | マウント | `addDays(today, 0..13)` の 14 日分を `useMemo` で生成 |
| 3 | 空き状況取得 | 選択日・施設・時間変更時 | `checkinApi.getAvailability({ location, facilityType, dates, startTime, duration })` を呼び出し、結果で「空き/残りわずか/満員」バッジを表示 |
| 4 | タブ切替「単日／複数日」 | `onClick` | `setMultiDateMode(enabled)`。複数日 OFF にした場合 `dates` と `recurringType` もクリア |
| 5 | 単日：日付タップ | `onClick` | `setDate(date)` |
| 6 | 複数日：日付チップ | `onClick` | `toggleDate(date)` で追加/削除。並び順は昇順ソート |
| 7 | 定期予約切替 | ラジオ | `setRecurring('WEEKLY' \| 'BIWEEKLY' \| null, count)`。選択時は基準日＋`interval * i` で日付配列を `setDates` |
| 8 | 開始時刻プルダウン | `onChange` | `setStartTime(time)`。候補は `LOCATION_TIME_SLOTS[location]`（ASP: 08-20 / YABASE: 07-20） |
| 9 | 利用時間プルダウン | `onChange` | `setDuration(hours)`。候補は `getAvailableDurations(startTime)`（`endHour <= 21` 制約） |
| 10 | 料金計算 | 条件変更時 | `useMemo` で `calculatePrice(location, facilityType, date, startTime, duration)` を再実行し `setTotalPrice` |
| 11 | 「確認画面へ」 | `onClick` | 必須項目全部揃っていれば `navigate('/payment')`。不足なら disabled |

**バリデーション**:
- 単日モード: `location && facilityType && date && startTime && duration`
- 複数日モード: `dates.length >= 1 && startTime && duration`
- 定期予約: `recurringType && recurringCount >= 1`（基準日必須）

#### 1.5.4 PaymentPage（`/payment`）確認・クーポン・決済

| # | イベント | トリガー | 処理内容 |
|---|---|---|---|
| 1 | 初期表示（必須チェック） | マウント | `location && facilityType && date && startTime && totalPrice` のどれか欠落で `/` に戻る |
| 2 | 会員割引の自動適用 | 価格・日数変更時 | `membershipApi.get()` → `discountType` に応じて 1 日あたり割引を算出し `setMemberDiscount(total, typeName)` |
| 3 | キャンセル時メッセージ | クエリ `?cancelled=true` | Stripe で戻ったときに「決済がキャンセルされました」を表示 |
| 4 | クーポンコード入力 | `onChange` | `couponInput` を更新（未検証） |
| 5 | 「適用」ボタン | `onClick` | `couponApi.validate({ code, location, totalPrice: totalPrice - memberDiscount })` → `setCoupon(code, discount)`。失敗時はメッセージ表示のみ |
| 6 | 「クーポンを外す」 | `onClick` | `setCoupon(null, 0)` + 入力欄クリア |
| 7 | 合計表示 | `totalPrice` / `couponDiscount` / `memberDiscount` 変更時 | `finalPrice = max(0, totalPrice - couponDiscount - memberDiscount)` を描画 |
| 8 | 「カードで支払う」 | `onClick` | `setIsLoading(true)` → `POST /api/payments/create-checkout` |
| 8a | 応答 `mode == "skip"` | - | `navigate('/complete?checkinId=...&groupId=...')` |
| 8b | 応答 `mode == "stripe"` | - | `window.location.href = checkoutUrl`（Stripe に遷移） |
| 8c | 409（重複/定員） | - | エラーメッセージを赤帯で表示。ボタン再活性化 |
| 8d | 500 等 | - | 汎用エラーを表示。ボタン再活性化 |
| 9 | デバッグスキップ | `debugStore` 参照 | `paymentEnabled == false` なら `skipPayment: true`、`remoteLockEnabled == false` なら `skipRemoteLock: true` をリクエストに付与 |

**リクエストペイロード（例）**:
```json
{
  "location": "ASP",
  "facilityType": "GYM",
  "date": "2026-05-01",
  "startTime": "10:00",
  "duration": 2,
  "couponCode": "WELCOME500",
  "skipPayment": false,
  "skipRemoteLock": false,
  "dates": ["2026-05-01","2026-05-08"],
  "recurring": { "type": "WEEKLY", "count": 2 }
}
```

## 2. API 設計

Vercel Hobby プランの関数上限 12 個の制約のため、関連機能を 1 本にまとめクエリ `type`/`action` で分岐している。

### 2.1 エンドポイント一覧（12 個）

| # | メソッド | パス | 概要 |
|---|---|---|---|
| 1 | GET/PUT | `/api/users/me` | 自ユーザー取得／更新 |
| 2 | GET | `/api/users/membership` | 自分の会員情報 |
| 3 | GET/POST | `/api/checkins` | 一覧・作成／空き状況／イベント・スクール一覧・申込 |
| 4 | GET/DELETE | `/api/checkins/[id]` | 詳細／キャンセル／領収書 PDF |
| 5 | POST | `/api/prices/calculate` | 料金計算 |
| 6 | POST | `/api/coupons/validate` | クーポン検証 |
| 7 | GET | `/api/member-types` | 会員種別一覧 |
| 8 | POST | `/api/reviews` | レビュー作成／取得 |
| 9 | POST | `/api/payments/create-checkout` | Stripe セッション発行 |
| 10 | GET | `/api/payments/stripe-callback` | Stripe 完了コールバック |
| 11 | POST | `/api/payments/stripe-webhook` | Stripe Webhook |
| 12 | GET/POST/PUT/DELETE | `/api/admin` | 管理系全般（`action` パラメータで分岐） |

### 2.2 認証方式

- **通常 API**: `Authorization: Bearer <LIFF_accessToken>`
  サーバー側で `https://api.line.me/v2/profile` に問い合わせて検証
- **Webhook** (`stripe-webhook`): `Stripe-Signature` ヘッダで署名検証
- **管理者 API** (`/api/admin`): LIFF トークン検証に加え、`ADMIN_LINE_USER_IDS`（カンマ区切り）に含まれることを確認

### 2.3 `/api/checkins`

#### POST `/api/checkins`（通常予約の DRAFT 作成）
```json
// Request
{
  "location": "ASP",
  "facilityType": "GYM",
  "date": "2026-05-01",
  "startTime": "10:00",
  "duration": 2,
  "totalPrice": 4400
}
// Response 201
{ "checkin": { "id": "...", "status": "PENDING", ... } }
```

#### GET `/api/checkins?type=availability&location=ASP&facilityType=GYM&dates=2026-05-01,2026-05-02&startTime=10:00&duration=2`
```json
// Response 200
{
  "2026-05-01": { "status": "available", "count": 0, "capacity": 1 },
  "2026-05-02": { "status": "full",      "count": 1, "capacity": 1 }
}
```

`status` は `available` / `few` / `full`。`few` は相席トレのときのみ（定員の 70% 超）。

#### GET `/api/checkins?type=events` / `?type=schools`
- `isActive == true` のみ。メモリで `date` / `createdAt` 順にソート。

#### POST `/api/checkins?type=event-register`
```json
{ "eventId": "evt_xxx" }
```
定員と重複申込を検証 → `eventRegistrations` に作成 → `events.currentCount` をインクリメント。

### 2.4 `/api/payments/create-checkout`

入力:
```ts
{
  location: LocationId;
  facilityType: FacilityType;
  // 単日: date / 複数日: dates / 定期: date + recurring
  date?: string;
  dates?: string[];
  recurring?: { type: 'WEEKLY'|'BIWEEKLY'; count: number };
  startTime: string;
  duration: number;
  couponCode?: string;
  skipPayment?: boolean;       // デバッグ用
  skipRemoteLock?: boolean;    // デバッグ用
}
```

処理シーケンス:
1. 認可 → ユーザー特定（無ければ自動作成）
2. 定期予約なら日付配列を展開
3. 重複チェック（定員1なら 1 件でも重なれば 409、相席は 10 件）
4. 各日ごとに `calculatePrice`
5. 会員種別の `discountType` で日ごとの割引額を算出
6. クーポン検証 → 会員割引後の合計に適用
7. 日数分の `checkins` を `status=PENDING` で作成（`groupId` で紐付け）
8. **skipPayment** なら即 RemoteLock 呼び出し→ PIN 設定→ `PAID` 更新
9. 通常は Stripe Checkout Session を作成、`paymentId` を各 `checkin` に保存

応答:
```json
{
  "checkoutUrl": "https://checkout.stripe.com/...",
  "checkinId": "...",
  "checkinIds": ["...", "..."],
  "groupId": "uuid-...",
  "mode": "stripe"  // or "skip"
}
```

### 2.5 `/api/payments/stripe-webhook`

- 受信: `checkout.session.completed`
- 処理:
  1. `metadata.checkinId` で Firestore の予約を取得
  2. `status == PAID` ならスキップ（冪等）
  3. RemoteLock に予約登録 → PIN 取得（失敗時はランダム 4 桁フォールバック）
  4. `status=PAID`, `pinCode`, `paymentId`, `updatedAt` を更新
  5. クーポン使用回数更新／`couponRedemptions` 記録

## 3. 処理ロジック詳細

### 3.1 料金計算（`src/lib/price.ts`, `api/prices/calculate.ts`, `api/payments/create-checkout.ts`）

3 箇所に同一ロジックが存在するため **同期が必要**（CLAUDE.md のメモリに記載）。

```ts
// 疑似コード
for (let i = 0; i < duration; i++) {
  const hour = startHour + i;
  const dayType = isWeekend(date) ? 'WEEKEND' : 'WEEKDAY';
  const dayPrices = PRICE_TABLE[location][facilityType][dayType];
  const price = 'ALLDAY' in dayPrices
    ? dayPrices.ALLDAY
    : dayPrices[hour < 17 ? 'DAYTIME' : 'EVENING'];
  total += price;
}
```

### 3.2 割引の適用順

```
通常料金
  ├─ 会員種別（FREE / PERCENTAGE / FIXED_PER_HOUR）→ 各日料金から差し引く
  └─ クーポン（PERCENTAGE / FIXED）             → 会員割引後の合計から差し引く
最終額 = max(0, 通常料金合計 − 会員割引合計 − クーポン)
※ クーポン割引は日数で按分し、剰余は先頭日に加算。
```

### 3.3 PIN 発行（RemoteLock 連携）

- 拠点・施設別に 1〜2 台のデバイスへ `/bookings` を POST
- ASP は **玄関 + 施設**（2 台）／やばせは **八橋ロック 1 台**
- 1 台目のレスポンス `pin` を採用し、2 台目以降は明示的に同じ `pin` を送信
- 複数日予約は 1 本目の PIN を `groupPinCode` として再利用（利用者は同じ PIN を使い回せる）
- API 失敗時: ランダム 4 桁生成（運用上は手動で PIN を再発行する）

### 3.4 重複予約チェック

- `checkins` から対象 `date` で取得し、`location` / `facilityType` / `status in (PENDING, PAID)` でメモリフィルタ
- `TRAINING_SHARED` は 10 件まで許容、それ以外は 1 件でも重なれば `409`

### 3.5 管理者判定（`server-lib/admin.ts` 相当）

- `/api/admin` エントリで LIFF トークン検証 → `profile.userId` を取得
- 環境変数 `ADMIN_LINE_USER_IDS`（カンマ区切り）に含まれていなければ 403

## 4. ログ仕様

### 4.1 フロントエンド (`src/lib/logger.ts`)

- `createLogger(scope)` で `{ debug, info, warn, error, op }` を返す
- 出力形式: `[LEVEL] HH:mm:ss.SSS [scope] message { ...payload }`
- 本番も `info` 以上を出力（Sentry 等に流す場合のフック地点）
- `op()` は操作ログ専用（接頭辞 `[OPERATION]`）

### 4.2 サーバー (`server-lib/logger.ts`)

- 1 行 JSON を `console.log` に出力 → Vercel ログで検索・集計可能
- フィールド: `ts`, `level`, `scope`, `msg`, 任意の `payload`
- `op(action, payload)` は `msg: "[OPERATION] <action>"` で出力

### 4.3 出力する主な操作ログ

| スコープ | action | タイミング |
|---|---|---|
| `App` | `app.boot`, `user.autoRegister.ok/fail` | 起動時 |
| `api.checkins` | `checkin.create` | DRAFT 作成 |
| `api.payments.create-checkout` | `checkout.skip.paid`, `checkout.stripe.session.created` | 決済準備 |
| `api.payments.stripe-webhook` | `webhook.paid` | 決済確定（※既存 `console.log` のまま） |
| `api.admin` | `admin.*` | 管理操作 |

> ログには `lineUserId` / `checkinId` / `sessionId` / 金額などを含めることで事後追跡を容易にしている。
